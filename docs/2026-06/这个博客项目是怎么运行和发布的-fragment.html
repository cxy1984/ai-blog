这篇文章用当前项目作为例子，解释从“写一篇 Markdown”到“在浏览器里看到博客”的完整流程。

我应该访问哪个 URL

现在有两个常用访问地址。

本地预览地址：

http://localhost:3003

线上 GitHub Pages 地址：

https://cxy1984.github.io/ai-blog

本地地址看的是你电脑上的文件，适合写文章时预览。

线上地址看的是 GitHub 仓库发布后的文件，适合分享给别人。

本地版本和线上版本有什么区别

可以把它们理解成两份不同位置的网站文件。

本地版本：

D:\code\ai-blog

线上版本：

GitHub 仓库 cxy1984/ai-blog 的 main 分支

你在本地写完文章以后，只有执行了：

git push

GitHub Pages 才能拿到最新文件并更新线上网站。

所以流程是：

本地写文章 -> 本地构建 -> 本地预览 -> git push -> GitHub Pages 发布

这个项目最核心的目录

真正需要长期关注的是这几个目录和文件：

markdown/       # 文章源文件，自己写的 Markdown 都放这里
docs/           # 构建生成的文章 HTML
pages/          # 构建生成的分页列表
overview.html   # 构建生成的文章概览和搜索数据
sitemap.xml     # 构建生成的站点地图
index.html      # 网站首页入口
template*.html  # 页面模板
src/            # 构建脚本

其中最重要的是：

markdown/

你写博客时，优先只改这里。

Markdown 是源文件

比如新建一篇文章：

markdown/2026-06/我的第一篇博客.md

内容类似：

# 我的第一篇博客

这里是正文。

第一行 # 标题 会被构建脚本识别为文章标题。

docs 是生成文件

执行：

npm run build

项目会把 Markdown 转成 HTML。

例如：

markdown/2026-06/我的第一篇博客.md

会生成：

docs/2026-06/我的第一篇博客.html
docs/2026-06/我的第一篇博客-fragment.html

其中：

  • .html 是完整文章页，适合搜索引擎和直接访问。
  • -fragment.html 是片段页，给首页的单页加载逻辑使用。

首页为什么会自动出现新文章

首页列表不是手写的。

执行：

npm run build

会运行：

src/generate_index_with_dates.js

这个脚本会扫描：

markdown/

然后按文章修改时间排序，生成：

pages/index.html
pages/index2.html
pages/index3.html
...

index.html 首页再加载这些分页列表，所以你写完 Markdown 并构建后，首页会自动出现新文章。

overview.html 是什么

overview.html 是文章概览页，也可以理解成文章索引页。

它由：

src/generate_overview.js

自动生成。

它会收集所有文章标题、链接、日期，然后写入页面里，方便搜索和浏览。

sitemap.xml 是什么

sitemap.xml 是给搜索引擎看的站点地图。

它由：

src/generate_sitemap.js

自动生成。

里面会列出站点里有哪些页面，例如：

https://cxy1984.github.io/ai-blog/index.html
https://cxy1984.github.io/ai-blog/docs/2026-06/我的第一篇博客.html

npm run build 到底做了什么

package.json 里定义了：

"build": "node src/convert_md_to_html_pandoc.js && node src/generate_index_with_dates.js && node src/generate_sitemap.js && node src/generate_overview.js"

也就是说，执行：

npm run build

等于依次执行 4 步：

1. Markdown 转 HTML
2. 生成首页分页列表
3. 生成 sitemap.xml
4. 生成 overview.html

这就是为什么你只写 Markdown，但网站的很多 HTML 文件都会变化。

npm run dev 是什么

npm run dev 是本地开发模式。

它会做三件事：

1. 清理并重新构建 docs
2. 启动文件监听
3. 启动本地 HTTP 服务

启动后访问：

http://localhost:3003

如果它一直开着，你就可以边写边预览。

GitHub Pages 怎么发布

GitHub Pages 的工作方式很简单：

读取 GitHub 仓库 main 分支里的静态文件，然后当成网站发布

你的仓库地址：

https://github.com/cxy1984/ai-blog

GitHub Pages 地址：

https://cxy1984.github.io/ai-blog

所以你本地改完以后，要执行：

git add .
git commit -m "add blog post"
git push

推送成功以后，GitHub Pages 才会更新线上网站。

为什么要删除原作者资源

这个项目最开始是从 GitHub copy 过来的，里面带了很多原作者的文章。

如果不删除，首页会混着显示别人的内容。

清理原则是：

保留项目框架和构建脚本
删除原作者文章源文件
删除原作者生成文件
保留自己的 Markdown
重新构建生成干净的网站

以后这个仓库就只显示自己的博客。

新建或编辑 Markdown：

构建：

本地预览：

http://localhost:3003

git commit -m "add blog post"

总结

这个项目不是动态博客系统，没有数据库，也没有后台管理页面。

它的本质是：

Markdown 源文件 + Node.js 构建脚本 + 静态 HTML + GitHub Pages

你负责写 Markdown。

脚本负责生成 HTML。

GitHub Pages 负责把 HTML 发布成网站。
